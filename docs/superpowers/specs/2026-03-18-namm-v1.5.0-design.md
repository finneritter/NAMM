# NAMM v1.5.0 Design Spec

## Overview

v1.5.0 addresses three UI bugs and adds a new **Chat Commands** feature — key-bound chat messages and slash commands executed with a single keypress.

---

## Bug Fixes

### Bug 1: Editor enabled toggle doesn't sync with macro list

**Symptom:** Toggling "Enabled" in the editor window doesn't update the macro list's toggle indicator.

**Root cause:** The editor toggle at `NammGuiScreen.java:1062` always calls `editingMacro.setEnabled()`, but when a profile is active, the macro list reads from `activeProfile.isMacroActive()` at line 252. These are separate state stores — the editor doesn't update profile state.

**Fix:** In the editor toggle handler (~line 1062), check if an active profile exists. If so, call `activeProfile.setMacroActive(macro.getName(), !activeProfile.isMacroActive(macro.getName()))` instead of `macro.setEnabled()`. This mirrors the logic already used in `handleMacroClick()` at lines 927-928.

**Files:** `NammGuiScreen.java` (~line 1062)

---

### Bug 2: KeyCaptureScreen shows blank (no prompt text)

**Symptom:** The trigger key capture screen opens but displays no text — just a blank screen.

**Root cause:** In Minecraft 1.21.11, `Screen.render()` no longer automatically renders a darkened background. The white/gray text is drawn over the raw game world and is effectively invisible.

**Fix:** Add `this.renderBackground(graphics);` before the `super.render()` call in `KeyCaptureScreen.render()` (line 23). This renders the standard dark translucent overlay, making the text visible. If `renderBackground` is not available in this MC version, use `graphics.fill(0, 0, this.width, this.height, 0xC0101010)` as a manual background fill.

**Files:** `KeyCaptureScreen.java` (~line 22-23)

---

### Bug 3: Editor window can't be un-minimized after collapsing

**Symptom:** Clicking the collapse arrow on the editor window minimizes it, but clicking the collapsed header doesn't expand it back. Close (X) still works.

**Root cause:** When the editor is collapsed, the `editorNameBox` widget (added via `addRenderableWidget` at line 1436) remains in the widget tree. Minecraft's `Screen` base class may be routing the click to the widget before the custom `mouseClicked` handler processes the header. Alternatively, the collapsed header click detection at line 787 starts a drag, but the `mouseReleased` toggle at line 905 may not fire if the click is consumed elsewhere.

**Fix approach:**
1. When collapsing the editor, temporarily remove `editorNameBox` from the widget tree (call `removeWidget`). Re-add it when expanding.
2. If that doesn't resolve it, add an explicit expand check: in `mouseClicked`, before the drag-start logic at line 787, check `if (editorCollapsed && isInHeader(...))` and immediately toggle `editorCollapsed = false; return true;` without going through the drag system.
3. Debug by adding logging to confirm which code path is actually executing on click.

**Files:** `NammGuiScreen.java` (~lines 787-804, 900-906, 1417-1437)

## New Feature: Chat Commands

### Data Model

New class: `com.namm.model.ChatCommand`

```java
public class ChatCommand {
    private String name;           // Display name, e.g. "Reset Totem Counter"
    private String message;        // Text to send, e.g. "/resetcounter" or "gg"
    private int triggerKeyCode;    // GLFW key code (-1 = unbound)
    private boolean triggerMouse;  // Whether trigger is a mouse button
    private boolean enabled;       // On/off toggle
}
```

Fields mirror `Macro`'s trigger fields for consistency. No playback modes, no steps — one key fires one message.

### Config / Serialization

- Add `chatCommands: List<ChatCommand>` field to `NammConfig` (alongside `macros` and `profiles`)
- Add `chatCommands` field to `MacroSerializer.ConfigWrapper`
- Add `chatWinX`/`chatWinY` fields to both `NammConfig` and `ConfigWrapper` for window position persistence
- Add corresponding getters/setters in `NammConfig`
- Null-guard in `MacroSerializer.load()`: `if (wrapper.chatCommands == null) wrapper.chatCommands = new ArrayList<>()`
- Export/import: Since `ConfigWrapper` already holds all data and Gson serializes it automatically, `exportToJson()` and `importFromJson()` will include chat commands by virtue of the `chatCommands` field being on `ConfigWrapper`. No signature changes needed — the existing `ConfigWrapper`-based serialization handles it. The `exportToJson(List<Macro>)` convenience method is only used for macro-specific export and does not need to change.
- The `MacroSerializer.save(List<Macro>, List<MacroProfile>, String, Path)` method is NOT used by `NammConfig.save()` — `NammConfig` builds a `ConfigWrapper` directly and calls `saveWrapper()`. So no changes to the `save()` method signature are needed; chat commands flow through the `ConfigWrapper` automatically.

**JSON format (in namm.json):**
```json
{
  "macros": [...],
  "profiles": [...],
  "chatCommands": [
    {
      "name": "Reset Totem Counter",
      "message": "/resetcounter",
      "triggerKeyCode": 294,
      "triggerMouse": false,
      "enabled": true
    }
  ],
  "chatWinX": -1,
  "chatWinY": -1
}
```

### Execution

Add a second loop in `TriggerKeyHandler.onClientTick()` after the macro loop (after line 73):

```
for each chatCommand in config.getChatCommands():
    if !enabled or triggerKeyCode == -1: skip
    detect rising edge (same pattern as macros, using previousKeyState with "cmd:" prefix)
    on rising edge:
        if message starts with "/":
            client.player.connection.sendCommand(message.substring(1))
            // Mapped name: ClientPacketListener.sendCommand(String) or
            // ClientPlayNetworkHandler.sendCommand(String) depending on mappings.
            // Verify exact method name against MC 1.21.11 Fabric mapped source.
        else:
            client.player.connection.sendChat(message)
            // Similarly: sendChatMessage(String) in some mappings.
```

- Chat commands are NOT profile-filtered (always available when enabled)
- No threading needed — `sendCommand`/`sendChat` are synchronous network calls
- Reuses the same `isKeyPressed()` helper and GLFW polling
- Key state tracked using a separate `Map<String, Boolean> previousChatKeyState` to cleanly avoid any collision with macro key state tracking

### UI: Fourth Window in NammGuiScreen

A new draggable, collapsible window following the exact same pattern as the Macro and Profile windows.

**Window properties:**
- Title: "Chat Commands"
- Width: `CHAT_WIN_WIDTH = 180` (slightly wider than macro window to fit command text)
- Default position: to the right of Profile window
- Collapse arrow, draggable header, position saved to config

**Window fields to add to `NammGuiScreen`:**
- `chatWinX`, `chatWinY` — position
- `chatCollapsed` — collapse state
- `chatScroll` — scroll offset
- `draggingWindow` values: extend to include `3` for chat window
- `editingChatCommand` — the chat command currently being edited inline (or null)
- `chatNameBox`, `chatMessageBox` — EditBox widgets for inline editing

**List rendering (follows macro window pattern):**
- Each row: enabled toggle indicator (left), command name (truncated), trigger key name (right)
- Hover highlight
- Separator lines between entries
- "+ New Command" button at bottom
- Right-click context menu: Edit, Delete

**Inline editing (when a command is selected for edit):**
The list entry expands into an edit form below the selected item (or at the bottom of the window):
- Row 1: EditBox for command name
- Row 2: EditBox for message text (the command/message to send)
- Row 3: Trigger key button (reuses `KeyCaptureScreen`) + Enabled toggle
- "Done" button to close inline editor

This avoids needing a separate editor window and keeps the chat commands self-contained.

**Integration into existing code:**
- `init()`: load `chatWinX`/`chatWinY` from config, create default position if -1
- `render()`: call `renderChatWindow()` after existing windows
- `mouseClicked()`: add chat window header check (draggingWindow = 3), content click handling
- `mouseReleased()`: add case 3 for collapse toggle and position save
- `mouseDragged()`: add case 3 for drag handling
- `mouseScrolled()`: add chat window scroll handling
- `onClose()`: save chat window position

### What Chat Commands Does NOT Include

- No profile integration (always available)
- No chaining (one key = one message)
- No delay or repeat
- No import/export of chat commands separately (they're included in the main NAMM export)
